# DESIGN.md — 설계 확정 문서

> 이 프로젝트는 **개발이 아니라 설계가 본질**이다. 코드는 언제든 다시 짤 수 있지만, 잘못된 스키마·UX 위에 쌓은 코드는 나중에 전부 갈아엎어야 한다. 그래서 스키마/화면 구조를 바꾸는 결정은 반드시 여기 먼저 기록하고, 컨펌받은 뒤에 코드에 반영한다.

## 진짜 문제 (개발은 이걸 위한 수단일 뿐)

**2026-07-08 개정 (2026-07-13 반영).** 기존 정의("통화 지식이 개인 기억에 갇힌다 → 팀 자산화")는 실제 통증이 아니었다. 진짜 통증은 **통화 중 인지 부담**이다.

위시켓 매니저는 프로젝트를 **검수하고 공고로 등록하기 위해** 의뢰인에게 전화한다. 그러려면 요구사항·예산·기간·기술 스택·검수 기준을 빠짐없이, 그것도 **좋은 질문으로** 끌어내야 한다. 그런데 통화하면서 동시에 "지금 뭘 더 물어야 하지"를 생각하는 건 에너지 소모가 크고, 놓치면 재통화 비용이 든다.

**그래서 이 시스템이 하는 일: 고객 말이 끝나면 "다음에 그대로 읽을 질문"을 화면에 띄운다.** 사용자는 뇌를 끄고 읽기만 한다. 기록·요약·팀 공유는 그 과정에서 자동으로 **남는 부산물**이지, 목적이 아니다.

- **핵심 가치**: 다음 질문 제안 (프로액티브 — 사용자가 요청하지 않아도 발화 종료 시 자동)
- **부산물**: 대화록·요약·오디오·팀 열람 (기존 스키마 그대로 유효)

**왜 범용 도구(Cluely 등)로는 안 되는가**: 범용 실시간 코파일럿은 이미 상품화돼 있고, 사용자가 물어보면 답하는 온디맨드 Q&A다. 하지만 "이 프로젝트를 공고로 올리려면 지금 뭘 물어야 하는가"는 **위시켓 도메인 지식**(검수 기준, 공고 필수 항목, 단가 감각, 6자리 프로젝트ID) 없이는 나오지 않는다. 이 시스템의 해자는 **도메인 프롬프트 + 용어집 + (추후) 과거 프로젝트 지식**, 그것 하나뿐이다. 은폐 기능·범용 Q&A는 전화 통화 맥락에서 무의미하므로 따라가지 않는다.

**성공 기준 (우선순위 순)**
1. **품질 (1차)** — 제안된 질문을 **수정 없이 소리내어 읽어도 자연스럽고, 실제로 물어볼 가치가 있는가.** 이게 안 되면 아무리 빨라도 0점이다.
2. **속도 (제약)** — 고객 발화 종료 → 제안 첫 글자까지가 **대화 흐름을 안 깨는 수준**(체감 2~3초 이내). 목표가 아니라 넘어야 할 바닥선.
3. **인지 부담** — 통화가 끝났을 때 "질문 짜내느라 지쳤다"는 느낌이 없는가.

**텔레프롬프터 원칙 (2026-07-03 확정 — 목적 개정 후에도 유효, 오히려 강화)**: 이 시스템의 끝은 통화 완전 자동화다. Phase 1에서 사람의 역할은 "AI가 써준 대사를 소리내어 읽는 것"뿐이고, Phase 2는 그 '읽는 사람'을 **내 목소리로 클로닝한 TTS**로 교체하는 것이다. 따라서:
- **제안은 조언("~라고 물어보세요")이 아니라 그대로 읽을 수 있는 완성 문장**이어야 한다.
- 제안 생성 파이프라인(맥락→다음 대사)이 곧 Phase 2의 두뇌다. Phase 2에서 갈아끼우는 건 출력부(화면→TTS)뿐이므로, 제안 엔진은 처음부터 "자동 통화의 뇌"라는 기준으로 만든다.
- **Phase 2 이주 계획**: 물리 IP 전화기 + 아날로그 탭은 Phase 1 한정. Phase 2에서는 클라우드 텔레포니(Twilio/SIP) + Vapi/LiveKit으로 전면 이주한다. **지금은 안 한다** — 실무 즉시 투입이 우선.

**도메인**: 위시켓 프로젝트 검수 · 공고 등록 상담 (요구사항 파악, 예산·기간 확정, 검수 기준 합의, 의뢰인 질문 응대)

**현재 사용자**: 본인 1인. **확정된 미래**: 팀 전체가 사용.

## 데이터베이스 — 최종 확정 스키마 (2026-07-03 컨펌)

**DB**: SQLite (`data/nophone.db`), `PRAGMA user_version`으로 스키마 버전 관리. 팀 확장 시 Postgres로 이전 예정이므로 **타입은 Postgres 호환만 사용**, ID는 **UUID**(중앙 서버 병합 시 충돌 방지).

### customers — 고객
```sql
id TEXT PK(uuid)
name TEXT
company TEXT
phone TEXT
memo TEXT
created_at TEXT
```
한 고객이 여러 번 전화하고 상담이 통화를 넘나들며 이어지므로, 통화를 고객 단위로 묶기 위해 필요. 발신번호는 하드웨어 구조상 자동 취득 불가 → 통화 후 화면에서 수동 태깅.

### calls — 통화 한 건
```sql
id TEXT PK(uuid)
customer_id TEXT FK(customers.id, nullable)   -- 태깅 전엔 NULL
agent TEXT                                     -- 담당자 (팀 확장 대비, 지금은 항상 본인)
started_at TEXT
ended_at TEXT
title TEXT
summary TEXT        -- AI 요약
briefing TEXT        -- 사전 브리핑 (통화 전 붙여넣은 프로젝트 설명) — 2026-07-03 추가, 청크 D4에서 반영
audio_path TEXT      -- 스테레오 wav (좌=고객/우=나) 원본. 2차(목소리 클로닝) 학습 재료 겸용
created_at TEXT
```

### utterances — 발화 한 줄 (실시간 자막)
```sql
id TEXT PK(uuid)
call_id TEXT FK(calls.id)
speaker TEXT        -- 'customer' | 'me'
text TEXT
ms INTEGER           -- 통화 시작 후 경과 ms
created_at TEXT      -- suggestions.created_at과 대조해 "그 시점까지의 대화" 역산용
```

### prompts — 프롬프트 프리셋 (화면 버튼이 여기서 생성됨)
```sql
id TEXT PK(uuid)
name TEXT            -- 버튼 이름 (예: "견적 초안")
template TEXT         -- 실제 프롬프트 내용
sort_order INTEGER
active INTEGER DEFAULT 1   -- 삭제 대신 비활성화 (과거 suggestions 참조 보존)
created_at TEXT
```
버튼을 코드가 아니라 데이터로 관리 → 프롬프트 추가/수정에 코드 변경 불필요.

### suggestions — AI 실행 기록 (통화 중 몇 번이든, 어떤 프롬프트로든)
```sql
id TEXT PK(uuid)
call_id TEXT FK(calls.id)
prompt_id TEXT FK(prompts.id, nullable)   -- 직접 입력이면 NULL
prompt_text TEXT     -- 실행 시점 프롬프트 원문 스냅샷 (프리셋이 나중에 바뀌어도 과거 기록 불변)
content TEXT          -- 결과
model TEXT
used INTEGER DEFAULT 0   -- 채택 여부 → 나중에 "어떤 제안이 실제 도움됐나" 분석 재료
created_at TEXT
```

### 검토 완료, 의도적으로 지금 안 만드는 것
- `projects`(프로젝트 단위 관리), `quotes`(구조화된 견적서 필드) — 실제 사용 데이터를 몇 주 쌓아본 뒤, 실사용 패턴에 맞춰 설계해야 정확함. 지금 만들면 추측 설계가 됨.
- 한국어 전문검색, 전사 오류 수정 이력 — 데이터가 쌓인 뒤의 문제.

### 오디오 보관 정책 (2026-07-03 확정)
- **전량 보관.** 통화당 스테레오 wav(좌=고객/우=나)를 `data/audio/`에 저장. 통화 1시간 ≈ 330MB.
- 근거: 녹취 증빙 + 재전사 가능성 + 2차(목소리 클로닝) 학습 재료. 용량 이슈가 실제로 생기면 그때 압축/정리 정책 추가.

## UI/UX — 최종 확정 설계 (2026-07-03 컨펌)

**시각 디자인(색상/타이포/컴포넌트)은 [STYLE.md](STYLE.md)를 무조건 따른다.** 아래는 화면 구조/정보 계층에 대한 결정이고, 실제 색·폰트·버튼 스타일은 STYLE.md 토큰으로 구현한다.

### 평가 기준 (통화 중이라는 맥락에서 도출)
1. 2초 파악 — 힐끗 봐서 핵심이 읽히는가
2. 조작 최소 — 모든 액션이 클릭/키 1번인가
3. 주의 계층 — 제일 중요한 게 제일 크게 있는가
4. 상태 표시 — 녹음/전사/AI가 돌고 있는지 안 읽어도 보이는가
5. 체감 속도 — 결과가 나오기 시작하는 게 빠른가
6. 통화중/통화후 분리 — 두 상황의 화면이 다른가
7. 실수 안전 — 잘못 눌러도 데이터가 안 죽는가
8. 확장성 — 프롬프트 증가/팀 합류에 화면이 버티는가

10회 검토 후 평균 4.0 → 8.8로 개선 (자세한 라운드별 기록은 대화 로그 참고, 핵심 변경만 아래 반영).

### 화면 1 — 통화 화면 (통화 중 유일하게 보는 화면) — 2026-07-03 개정 (화면분할+탭)
```
┌────────────────────────────────────────────────────────────────┐
│ ● 통화중 12:34   전사연결됨              [통화 종료]            │
├────────────────────┬───────────────────────────────────────────┤
│ 실시간 대화록 (40%) │ 브리핑: 전원 연계 방송 플랫폼 · 예산 3천…   │
│                    │ ┌───────────────────────────────────────┐ │
│ 나    1단계 예산이..│ │ 💬 다음 멘트 (그대로 읽는 완성 대사)      │ │
│ 고객  네, 3천만...  │ │ "그러면 1단계는 3천만 원 상한 기준으로…" │ │
│ 전사중인 말은 회색..│ │ 직전: "혹시 예산은 어느 정도로…" (흐림)  │ │
│   ↓ 자동 스크롤    │ └───────────────────────────────────────┘ │
│                    │  [⌥1 견적초안] [⌥2 절차] [⌥3 요약] [✏️직접]│
│                    │ ┌─────────────────┬─────────────────────┐ │
│                    │ │ 📋 요구사항(상시) │ 📄 실행 결과 (탭 전환)│ │
│                    │ │ 자동 갱신되는     │ 견적/절차/요약/직접이  │ │
│                    │ │ 살아있는 문서     │ 여기 표시 [복사]      │ │
└────────────────────┴─┴─────────────────┴─────────────────────┴─┘
```
개정 이유: "요구사항이 나와야 견적이 나온다" — 요구사항은 버튼 결과가 아니라 **통화 내내 자동 갱신되는 상시 패널**로 승격. 견적을 뽑는 순간 옆에 요구사항이 항상 보인다. 나머지 실행 결과(견적/절차/요약/직접입력)는 우측 하단에서 탭으로 전환.

핵심 결정:
- **제안 영역이 주인공, 대화록은 참조용** (원안은 반대였음 — 실제 필요한 건 "다음에 뭐라 말할지")
- **멘트 제안 = 그대로 읽는 완성 대사** (텔레프롬프터 원칙). 최신 제안은 크게, **직전 제안 1개는 아래 작고 흐리게** 유지 — 좋은 제안이 다음 제안에 밀려 사라지지 않게. 전체 히스토리는 DB에 있으므로 기록 화면에서 열람.
- **요구사항 정리는 상시 패널** (버튼 아님) — 통화 내내 자동 갱신. 요구사항용 내장 프롬프트(builtin-requirements)를 별도로 둔다. 버튼은 견적/절차/요약/직접입력 4개.
- **사전 브리핑**: 통화 시작 시 프로젝트 설명(위시켓 등록 문서 등)을 붙여넣는 입력칸. AI가 통화 시작부터 맥락을 알고 제안 — RAG(1.5차)의 전 단계. 통화 중에는 우측 상단에 한 줄 요약으로 표시.
- 고객 태깅·제목 수정 등 통화 후에만 필요한 조작은 **이 화면에 안 넣음** (통화 중 타이핑 강요 금지)
- 버튼 실행에 **단축키(⌥1~⌥9, Option+숫자)** — 통화 중 마우스 이동 최소화. ⌘1~9는 맥 브라우저가 탭 전환으로 선점해서 웹페이지가 가로챌 수 없으므로 ⌥ 사용 (2026-07-03 검증에서 발견·수정)
- 자동 멘트 제안과 버튼 실행 결과는 **물리적으로 분리된 영역** (하나 읽는 중 다른 게 덮어쓰지 않도록)
- 상단 상태바(● 녹음중 + 경과시간) + 발화 감지/전사 진행이 눈에 보이게 → "살아있음"이 안 읽어도 보임
  - 정확한 동작(STT가 발화 종료 후 전사하는 방식이므로): 상대가 말하는 동안엔 `🎤 말하는 중...` 표시 → 말이 끝나면 전사 텍스트가 회색으로 스트리밍 → 완료되면 검정 확정 문장으로 전환 (2026-07-03 검증에서 정밀화)
- 새 제안 도착 시 은은한 하이라이트 점멸 (소리 알림 금지 — 통화 중이므로)
- 통화 종료만 확인 1단계, 나머지 버튼은 여러 번 눌러도 무해

### 화면 2 — 기록 화면 (통화 후 / 평상시)
- 통화 목록 (날짜·고객·제목·시간) → 클릭 시 상세: 대화록 전문, 그때 실행한 AI 기록, 오디오 재생, **고객 지정(태깅)**, 제목·요약 수정

- 실사용 전 검증 불가한 부분은 실제 통화로 써보며 조정.

## 도메인 컨텍스트 (용어집) — 2026-07-03 추가

전화 협대역 음질에서 고유명사(회사명 등)가 자주 오인식됨 (예: "위시켓"→"미식기"). 대응:
- **`glossary.txt`**: STT가 정확히 알아들어야 할 고유명사·용어를 한 줄에 하나씩. STT 세션의 `prompt`로 주입해 인식률을 높인다. `glossary.example.txt`를 복사해서 사용하고, 실제 파일은 gitignore(고객사명 등 민감정보 가능성).
- **STT 모델**: `gpt-4o-transcribe`로 확정 (mini는 용어집을 넣어도 "위시켓"을 못 잡음, 상위 모델은 잡음). 비용 분당 ~8원, 정확도 우선.
- 이 용어집은 향후 AI 제안(청크 D·E)의 시스템 컨텍스트로도 재사용 예정 → Phase 1.5 RAG의 씨앗.

## 스택 (확정)

Python 3.12 + uv · sounddevice (오디오 캡처) · OpenAI 스트리밍 전사 API (STT, 교체 가능하게 모듈 분리) · OpenAI GPT 스트리밍 (AI 제안) · FastAPI + WebSocket · 단일 HTML + 바닐라 JS · SQLite
