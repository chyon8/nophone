# PLAN.md — 개발 계획

> 설계 원본: [DESIGN.md](DESIGN.md) · 시각 디자인: [STYLE.md](STYLE.md) · 작업 규칙: [CLAUDE.md](CLAUDE.md)
> 진행 방식: 청크 하나 완료 → 결과 보고 + 사용자 확인 체크리스트 → 컨펌 → git 복붙 블록 제공 (사용자가 직접 실행) → 다음 청크. **이 순서를 벗어나지 않는다.**
> 상태 표기: ✅ 완료 · 🔄 진행 중 · ⏳ 대기

## 의존성 그래프 (순서를 정한 근거)

```
[하드웨어 ✅] → [캡처 ✅] → [STT ✅]
                                │
[A. DB] ──────────┬─────────────┤
                  │             ▼
                  │      [B. 서버+통화세션] ← 캡처/STT를 세션으로 묶음, DB에 발화 저장
                  │             │
                  │             ▼
                  │      [C. 통화 화면 UI] ← 자막이 흘러야 UI 검증 가능
                  │             │
                  ├──────►[D. AI 멘트 제안] ← 대화록(B)과 화면(C)이 있어야 의미 있음
                  │             │
                  └──────►[E. 프롬프트 버튼] ← prompts 테이블(A) + 제안 엔진(D)의 컨텍스트 재사용
                                │
                         [F. 통화 마무리] ← 세션(B)의 시작-끝 흐름 완성
                                │
                         [G. 기록 화면] ← 저장된 데이터(A~F)가 있어야 보여줄 게 있음
                                │
                         [H. 실전 테스트] ← 전부 합쳐진 뒤에만 가능
```

- **A(DB)가 최우선**인 이유: B(발화 저장), D·E(제안 저장, 버튼=prompts 테이블), G(조회) 전부가 A에 의존. A가 흔들리면 전부 재작업.
- **C(UI)가 D(AI)보다 먼저**인 이유: 자막이 화면에 흐르는 걸 눈으로 확인해야 D의 제안 타이밍·표시를 검증할 수 있음.
- **E는 D 다음**인 이유: 버튼 실행은 D에서 만드는 "컨텍스트 어셈블러"를 그대로 재사용. 순서를 바꾸면 같은 걸 두 번 만듦.
- **G는 후반**인 이유: 통화 중 화면이 핵심 가치이고, 기록 화면은 쌓인 데이터가 있어야 검증 가능.

## 리스크 관리

| 리스크 | 대응 | 확인 시점 |
|---|---|---|
| 사이드톤 중복 전사 (내 목소리가 고객 채널에 섞임) | 채널 간 유사 문장 감지·제거 로직 | H1에서 실측 후 (파일 모드로는 재현 불가) |
| 장통화(30분+) 시 토큰 비용·지연 증가 | v1은 최근 N발화만, 실측 후 요약 압축 추가 | D1 설계에 반영, H1에서 실측 |
| STT 지연 변동 (API 서버 상태) | 재연결 로직 + 상태바에 이상 표시 | B2, C1 |
| 브라우저 탭 가려짐 | 1차는 운용으로 수용, 추후 항상-위 미니창 | Phase 1 이후 재검토 |
| SQLite 동시 접근 | 1차는 단일 서버 프로세스만 접근하므로 문제없음. 팀 확장 시 Postgres 이전 | Phase 1.5 이후 |

## 현재 위치

```
✅ 하드웨어  ✅ 캡처  ✅ STT  ✅ 설계 확정  ✅ 청크 A (DB)  ✅ 청크 B (서버+세션, M1)
→ 다음: 청크 C (통화 화면 UI — STYLE.md 정식 적용)
```
