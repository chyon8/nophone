## 의존성 그래프 (순서를 정한 근거)

```
[하드웨어 ✅] → [캡처 ✅] → [STT ✅]
                                │
[A. DB] ──────────┬─────────────┤
                  │             ▼
                  │      [B. 서버+통화세션] ← 캡처/STT를 세션으로 묶음, DB에 발화 저장
                  │             │
                  │             ▼
                  │      [C. 통화 화면 UI] ← 자막이 흘러야 UI 검증 가능
                  │             │
                  ├──────►[D. AI 멘트 제안] ← 대화록(B)과 화면(C)이 있어야 의미 있음
                  │             │
                  └──────►[E. 프롬프트 버튼] ← prompts 테이블(A) + 제안 엔진(D)의 컨텍스트 재사용
                                │
                         [F. 통화 마무리] ← 세션(B)의 시작-끝 흐름 완성
                                │
                         [G. 기록 화면] ← 저장된 데이터(A~F)가 있어야 보여줄 게 있음
                                │
                         [H. 실전 테스트] ← 전부 합쳐진 뒤에만 가능
```

- **A(DB)가 최우선**인 이유: B(발화 저장), D·E(제안 저장, 버튼=prompts 테이블), G(조회) 전부가 A에 의존. A가 흔들리면 전부 재작업.
- **C(UI)가 D(AI)보다 먼저**인 이유: 자막이 화면에 흐르는 걸 눈으로 확인해야 D의 제안 타이밍·표시를 검증할 수 있음.
- **E는 D 다음**인 이유: 버튼 실행은 D에서 만드는 "컨텍스트 어셈블러"를 그대로 재사용. 순서를 바꾸면 같은 걸 두 번 만듦.
- **G는 후반**인 이유: 통화 중 화면이 핵심 가치이고, 기록 화면은 쌓인 데이터가 있어야 검증 가능.

## Phase 1 — 통화 코파일럿 (1차 범위)

### 🔴 최우선 검증 — STT (2026-07-07, 오늘 목표)
> **이 서비스 전체가 STT의 정확도·실시간성에 얹혀 있다.** 자막이 부정확하면 그 위의 AI 제안·견적·자동통화가 전부 무의미. 따라서 UI(청크 C-2)를 잠시 멈추고, "한국어 전화 음성이 실시간으로 충분히 정확하게 전사되는가"를 먼저 증명한다. **이게 검증돼야 서비스를 계속 만든다.** STT 모듈은 교체 가능(룰 1)하므로 엔진을 바꿔가며 테스트한다.
>
> **엔진 후보**: OpenAI `gpt-4o-transcribe`(즉시 가능) · Return Zero RTZR(한국어·콜센터 특화, 키 필요) · Clova(NCP 필요). Whisper(배치)·GPT-realtime(음성대화=Phase2용)은 이 용도 부적합.

| 항목 | 내용 |
|---|---|
| V1 | **오프라인 정확도**: 샘플 녹취 전체 전사 → 핵심어(금액·기간 등 숫자, 고유명사, 6자리 프로젝트ID) 정확도 검토. mini vs full, 용어집 on/off 비교. 측정 도구(`stt_eval.py`) 작성 |
| V2 | **실시간 지연 측정**: 발화 종료 → 텍스트 확정까지 지연(ms)을 파일 1배속에서 측정 |
| V3 ✅ | **실제 통화 검증 (2026-07-07 완료)**: 실통화에서 고객 채널이 실시간·정확하게 전사됨 확인. "나" 채널 다듬기(맥북 마이크 환각·헤드폰 마이크 전기탭·사이드톤 중복)는 곁가지로 뒤로 미룸 |
| V4 ✅ | **제안 LLM 결정 (2026-07-13)**: `suggest.py`로 발화종료→첫글자 실측. **`gpt-4.1` 채택** — 속도·품질 둘 다 1위. gpt-5 계열은 느리고(첫토큰 1.4~2.3s) gpt-5-mini는 괄호 나열식 만연체라 **소리내어 읽기 부적합**(텔레프롬프터 원칙 위반). STT 엔진 교체(RTZR 등)는 **불필요 — 보류**: 현행 OpenAI cascade로 이미 기준 통과 |
| | **실측 (LLM 구간, 워밍업 후 2회 평균)**: gpt-4.1 **570ms** · gpt-4.1-mini 780ms · gpt-5(minimal) 1440ms · gpt-5-mini(minimal) 1490ms · gpt-5-mini(low) 2340ms |
| | **실측 (end-to-end, 발화종료→제안 첫글자)**: 60초 샘플 3발화 · **평균 1486ms** · 중앙 1322ms · 최대 1876ms → DESIGN.md 속도 기준선(2~3초) **통과** |
| **성공 기준** | 실통화에서 숫자·고유명사가 신뢰할 만하게 잡히고, 발화종료→텍스트 지연이 대화 흐름을 해치지 않는 수준 |

## 📌 나중에 (사용자 리마인드 요청 — 안 까먹게)

- **통화 시작 시 맥락 입력 (사전 브리핑 D4의 확장)**: 텍스트뿐 아니라 **파일 / URL**도 붙여넣어 맥락으로 주입. **프로젝트 ID는 6자리 숫자**(예: 153545) — 위시켓 프로젝트 식별자. ID로 등록 문서를 자동으로 끌어오는 것도 검토. 지금 구현 안 함, 사전 브리핑 만들 때(D4) 사용자에게 다시 알릴 것.
